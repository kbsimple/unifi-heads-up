| process.env → src/lib/unifi/client.ts | Server-only env vars consumed at function entry; never reach the client bundle |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-6-01 | Tampering | undici Agent TLS configuration | mitigate | `rejectUnauthorized: false` appears in EXACTLY ONE place — the module-level `new Agent({ connect: { ... } })` — and is passed via `dispatcher: agent` only to fetches targeting `${UNIFI_HOST}`. Acceptance criteria forbid `NODE_TLS_REJECT_UNAUTHORIZED` and require exactly one `new Agent(`. HIGH severity — blocking. |
| T-6-02 | Information Disclosure | UNIFI_API_KEY in client bundle | mitigate | `import 'server-only'` at the top of client.ts (preserved from v1) — Next.js build fails if any client component imports this file. UNIFI_API_KEY has no `NEXT_PUBLIC_` prefix. MEDIUM severity. |
| T-6-03 | Tampering | UNIFI_HOST env var | accept | UNIFI_HOST is server-side only and never user-supplied; set at deploy time by the operator. The risk of an attacker modifying it requires shell access to the host (out of scope for app threat model). LOW severity — documented as accepted. |
| T-6-04 | Denial of Service | UniFi console request timeout | mitigate | `AbortSignal.timeout(10_000)` on every fetch — same 10s budget as v1, prevents hanging requests from blocking the Next.js request handler. LOW severity. |
| T-6-05 | Information Disclosure | Error messages | accept | `'UniFi API error: 403 Forbidden'` etc. surface backend status. Acceptable for a household-internal app on LAN — error boundaries in callers prevent leakage to UI. LOW severity. |
| T-6-06 | Repudiation | Firewall toggle action | accept | No audit log in v2.0. Out of scope per requirements (deferred). LOW severity. |
